# AutoQAGBenchmark 全流程操作手册：从第一性原理到每一步

> 本文目标：用一条逻辑主线，把"我们到底要做什么 → 题目怎么定义 → 数据基座为什么这样建 →
> 子图怎么协同出题 → 怎么自更新迭代"串成一个连贯整体。每一步都给具体例子，全部用自然语言
> 和少量数学式说明，不放代码。读完应能回答三个核心问题：
> **①图谱怎么构建（含哪些图、哪些点、为什么）；②QA 怎么建立（先 Q 还是先 A，种子怎么初始化，子图怎么扩展）；③怎么自更新和出题。**

贯穿全文用同一个**走查样例**（材料/钙钛矿太阳能电池领域）：

- 论文 **P1**：正文有一句 `"The PCE of device A reaches 23.5% under 85°C aging condition."`；
  含 **Table 1**（行：device A / device B；列：PCE (%)、Voc (V)）；含 **Figure 3**，图注提到 efficiency。
- 论文 **P2**：另一篇钙钛矿论文，报告某器件 `PCE = 21.8%`。

## 0. 第一性原理：我们在生产什么，约束从哪来

我们要为**专用科研领域**自动生产一个 **benchmark（评测题库）+ 训练语料**。一道合格的科研题，必须同时满足三条**硬约束**，整套系统就是从这三条倒推出来的：

| 硬约束 | 含义 | 它倒逼出的设计 |
|---|---|---|
| **可追溯** | 每个答案都能指回原文 `⟨paper_id, section, chunk_id, span⟩` | 点必须带物理地址 → **物理证据图 G0** |
| **可验证** | 数值/单位/条件/证据能被机器逐项核对 | 点要**分类型**（数值是数值、条件是条件）→ **Schema 化分层** |
| **难度可控且非伪多跳** | "难"是真的要跨多段证据，而不是句子写长 | 难度用**结构变量**定义；出题前先做**逻辑规划** → **语义覆盖层 Ωq** |

一句话：**因为要可追溯，所以建带地址的图；因为要可验证，所以点要分层分类型；因为难度要真实，所以出题不是"采点拼句"而是"先规划逻辑再回填证据"。** 下面每一节都服务于这三条。

整条流水线 10 个模块，对应六个阶段：

```
A 解析归一化   ingest → parse → normalize        ⇒ evidence_blocks.jsonl（带地址的证据块）
B 数据基座 G0  graph                              ⇒ nodes.jsonl + edges.jsonl（固定物理图）
C 语义规划     semantic_plan                      ⇒ question_plans.jsonl（每题一份规划）
D QA 落地      generate                           ⇒ qa.jsonl + corpus/
E 验证修复闭环 corrupt → verify → repair          ⇒ violations.jsonl（违规）→ 修复
F 导出         output                             ⇒ benchmark.jsonl + stats.json
```

## 1. 先把"题"定义清楚（这一步决定后面一切）

出题之前，必须先回答："一道题由什么构成？什么叫难？" 这两个定义是后面"数据基座该长什么样""子图该扩展到多大"的需求来源。

### 1.1 题目 = 一组**逻辑角色**被填满（不是随机问一句）

每个题型在系统里被定义成一组**必需的推理角色**（`required_reasoning_roles`）。题目"成立"的充要条件是：这组角色都能在图里找到真实节点来承担。

以**比较题**为例，它的完整角色 schema 是：

```
object_A, object_B, shared_attribute, value_A, value_B,
unit, aligned_condition, evidence_A, evidence_B, comparison_criterion
```

其中**最小角色集**（硬门槛，缺一不可）是：
`object_A · object_B · shared_attribute · value_A · value_B · evidence_A · evidence_B`。

> **例**：要出"device A 和 device B 在 PCE 上谁高？"——
> `object_A=device A`、`object_B=device B`、`shared_attribute=PCE`、`value_A=23.5%`、`value_B=表中B的PCE`、
> `evidence_A/B=各自所在的 chunk/表格`。少了 `value_B`，这题就不允许生成。

系统当前支持 10 类题型（`atomic / numerical / condition / comparative / mechanism / table / formula / multi_hop / cross_paper / summary`），每类都有自己的角色 schema 与种子策略。**"先定角色、再找节点"是后面整个规划的纲领**。

### 1.2 难度 = **推断算子下限 ⊕ 结构变量封顶**，不是语言长度

难度 L1–L4 由两件事**共同**决定——**这道题要求的推断操作有多难（推理下限）**，以及**要跨多少真实证据（结构上限）**。两者各管一头、互不淹没：

**① 结构分（检索/组合难度）**，公式（自然语言版）：

```
结构难度分 = 证据点数 + 路径长度 + 模态数 + 2×跨章节距离 + 约束数(单位/条件) + (跨文献?+3)
```

- 跨文献 / 跨章节 / 跨模态（≥2 种）/ 或难度分 ≥9 → **L4**；≥6 → **L3**；≥3 → **L2**；否则 **L1**

**② 推断算子下限（推理难度）**：每道题携带一个**推断算子**（见 §3.6，如比较=COMPARISON、跨文献=CROSS_PAPER_ALIGN、机制=CAUSAL_CHAIN），每个算子有一个**推理难度下限** `INFERENCE_FLOOR`（如 CROSS_PAPER_ALIGN=L3、CONDITION_SHIFT=L3、CAUSAL_CHAIN=L3、COMPARISON=L2、ATTRIBUTE_LOOKUP=L1）。

**③ 合成**：`难度 = max(结构分, 算子下限)`，**再按真实跨 chunk 数封顶**（§3.7，单 chunk≤L2、2 chunk≤L3、≥3 才 L4）。

```
难度 = cap_by_chunk( max(结构难度等级, 推断算子下限) )
```

> **为什么要"下限 ⊕ 封顶"两步？** 因为难度有**两个正交维度**:检索难（要跨多少 chunk/paper 才找到证据）和推理难（要做哪种推断操作）。早先把它们塞进一个标量和,会让"3 篇论文各 1 跳"(检索难、推理trivial)和"单 chunk 4 步推导"(推理难、检索trivial)塌缩成同一标签。现在:**算子下限**保证跨文献题即使结构上只跨 2 chunk 也不被低标为 L2;**跨 chunk 封顶**保证单 chunk 富节点即使节点很多也压到 L2——检索与推理各自计分。
>
> **为什么不用句子长度？** 因为"句子长"可以靠啰嗦堆出来，是**伪难度**；而"要同时用到 device A 的表格值 + device B 的表格值 + 85°C 条件 + 跨两个 chunk + 做一次跨对象比较推断"才是**真难度**。难度因此被锚定在"必须做哪种推断 + 必须跨多少真实证据"上。

每个难度还规定了**最低结构要求**（跨 chunk 数, 证据 span 数, 物理路径长）：

```
L1=(1,1,0)  L2=(1,2,1)  L3=(2,3,2)  L4=(3,4,3)
```

这张表后面会变成**硬验收线**：一道标了 L3 的题，子图若只覆盖 1 个 chunk，就会被判伪多跳并降级（见 §3.6）。

> **衔接**：§1 告诉我们——题目需要"分类型、带证据、能跨段"的节点。这正是 §2 数据基座必须提供的东西。

## 2. 数据基座 G0：为什么这样建、怎么建（创新①）

### 2.1 为什么是"图"，而且是"两张耦合的图"

科研问题天然是关系问题（某对象在某条件下某指标的某值，并被某图表支撑）。要让这些关系可计算、可追溯，就需要图。

#### 2.1.0 什么叫"平铺 KG"，它为什么有两个死穴

**平铺 KG（flat knowledge graph）** 指传统那种"只有一层、只存 `实体—关系—实体` 三元组、且把所有论文的三元组揉进同一个大池子"的知识图谱。它的特征是：节点只有"实体"一种身份，边只有"语义关系"一种用途，**既不记录这条知识来自哪里，也不保留它在文章里的组织结构**。例如把全语料压成一堆：

```
(device A, has_PCE, 23.5%)
(device B, has_PCE, 21.8%)
(PCE, unit, %)
...
```

看似干净，但出科研题时会暴露**两个死穴**：

- **死穴一 · 丢来源（无 provenance）**：三元组 `(device A, has_PCE, 23.5%)` 不知道这个 23.5% 出自**哪篇论文、哪一节、哪个 chunk、哪句原文**，更不知道它的**适用条件是"85°C 老化"**。于是答案无法指回原文 span，无法被验证器核对，"可追溯"这条硬约束直接失守——而且 `85°C aging` 这种条件在平铺三元组里往往被当成噪声丢掉，出来的题会过度泛化（"device A 的 PCE 是 23.5%"，省掉了条件）。
- **死穴二 · 丢结构（无 structure）**：把所有论文揉进一个池子后，**文章的层级（Paper→Section→Chunk）没了、模态归属（这个值来自正文还是 Table 1 还是 Fig 3）没了、跨文献边界（这是 P1 的还是 P2 的）也糊在一起**。结果：① 无法按"章节/段落"定位证据，难度也就无法用"跨几个 chunk / 跨几节"来刻画——伪多跳防不住；② 同名实体跨文献混为一谈，分不清"P1 的 PCE"和"P2 的 PCE"，跨文献题无从谈起。

> 一句话：平铺 KG 把"知识"抽象得只剩裸三元组，**代价是把"它从哪来、在原文怎么组织"这两类元信息全丢了**——而这恰恰是科研出题最需要的两样东西。

#### 2.1.1 我们的 KG 不是平铺的：靠三件事把"来源"和"结构"补回来

我们的 G0 **不平铺**，区别就在于把平铺 KG 丢掉的两样东西显式建进图里：

1. **每个点带物理地址**（补回"来源"）：节点不是裸实体，而是带 `⟨paper_id, section_path, chunk_id, span⟩` 的结构单元 + `polarity/confidence`。`23.5` 这个 ValueNode 知道自己来自 P1 / results 节 / chunk-007 / 哪句话，并经 `under_condition` 挂着 `85°C aging`。
2. **点天然分层、边带"建边依据"**（补回"结构"）：节点分定位层/高层语义/中层约束/底层证据四层（见 §2.2），每条边都记 `build_reason`（`document_structure` 还是 `physical_cooccurrence`），所以"文章结构"和"语义共现"两类关系不会糊在一起。
3. **两张耦合子图而非一个池子**（同时补"来源 + 结构"）：用一张**纵向结构图**保留文章组织、一张**横向共现图**承载跨对象/跨文献关联，并保留 `paper_id` 边界（跨文献只经显式 `same_as` 连接）。

于是 G0 由**两张相互耦合的子图**组成：

- **纵向 · 文章结构导航图**：`Paper → Title / Section → Chunk → 点`，边类型 `contains`，依据是**文章自然结构**。
  作用：单篇文献的**定位与可追溯**——任何一个点都能顺着 `contains` 走回它所在的 chunk、section、paper。
- **横向 · 物理共现图**：在**同句/同段/同 chunk/同表格行列/同图注**内的点之间建边，依据是**物理共现**。
  作用：**跨对象、跨文献的语义关联**——支撑比较、机制、跨文献等复杂题。

**关键设计：边的语义类型 = f(两端点的标签)，边是否建立 = 是否物理共现。** 前者是查表（规则），后者是物理事实——**都不依赖 LLM 去"想象"关系**，从源头杜绝幻觉边。

### 2.2 点的"自然分层"——分层不是事后加的，是抽取时天生的

点在抽取那一刻就带着层级，因为类型本身就分层：

| 层 | 节点类型 | 例（走查样例） |
|---|---|---|
| 定位/结构层 | Paper / Title / Section / Chunk | P1、Title、`Results` section、chunk-007 |
| 高层语义层 | Concept / Method / Claim | device A、aging test、"A 在高温下更稳定" |
| 中层约束层 | Attribute / Condition | PCE、85°C aging |
| 底层证据层 | Value / Unit / Figure / Table / Equation / Caption / Evidence | 23.5、%、Table 1、Fig 3 caption |

> **为什么分层有用（理解难点）**：分层让"一条证据链"有了天然的上下走向——
> 高层 `device A` → 中层 `PCE` → 底层 `23.5` → `%`。出题时**向下走**能取到完整证据链，
> **向上走**能确认"这个 23.5 到底是谁的、在什么条件下的"，避免数值悬空。分层正是 §3 子图扩展的轨道。

### 2.3 建图三步走（带例子）

**第一步 · 建结构图（纯规则，无 LLM）。** 遍历证据块，建 `Paper→Section→Chunk` 的 `contains` 骨架；图/表/公式块挂到所属 Section，图注挂到所属图/表。

> 例：P1 建出 `P1::PAPER → P1::SEC::results → chunk-007`，Table 1 挂到 results，Fig 3 的 caption 挂到 Fig 3。

**第二步 · 抽点 + 建共现边。** 这是**唯一**用到 LLM 的地方，而且 LLM 只做一件事——**给文本块里的点打类型标签 + 标出哪些点物理共现**，输出形如 `("point"<|>PCE<|>Attribute<|>...)`、`("point"<|>23.5%<|>Value<|>...)`。

拿到点之后，**边怎么连不问 LLM**：

- 查表：`(Attribute, Value) → has_value`、`(Value, Unit) → has_unit`、`(Condition, Value) → under_condition`、`(Concept, Attribute) → has_attribute` …方向也由表规范化。
- 表格更不需要 LLM：列头=`Attribute`、行标签=`Concept`、单元格=`Value`，按行列结构直接建 `has_value` / `same_table_row` / `same_table_column`，单位从表头括号里抽。

> 例（那句话）：LLM 标出 `device A(Concept) / PCE(Attribute) / 23.5(Value) / %(Unit) / 85°C aging(Condition)`；
> 规则随即连出 `device A —has_attribute→ PCE —has_value→ 23.5 —has_unit→ %`，并把 `85°C aging —under_condition→ 23.5`。
> 这条链就是这道题未来的"答案证据骨架"。

**第三步 · 跨模态 + 跨文献边（纯规则）。**

- **跨模态引用**：正文出现 "Fig. 3" / "Table 1" → 该 chunk `references` 对应 Figure/Table 节点。
- **跨模态对齐**：图注里的 `efficiency` 与正文里的 `PCE/efficiency` 同名（同 `canonical_id`）且分处不同模态 → 建 `aligns_with`，把"同一对象的图注证据与正文证据"并到一起。
- **跨文献聚合**：P1 的 `PCE` 与 P2 的 `PCE` 同 `canonical_id` 但不同 `paper_id` → 建 `same_as`。这条边就是未来**跨文献题**的物理依据。
  > **注意（条件兼容性）**：`same_as` 只表示"同一指标/概念",**不等于"可比"**。P1 的 23.5% 限定在 `85°C 老化`、P2 的 21.8% 若条件不同或未知,直接比大小是**伪可比**。因此 `same_as` 只建立"对齐候选",真正能否出比较/跨文献题,要在规划阶段过**条件兼容性门**(§3.5 / §3.6):两端的 `under_condition` 同维度须同值或可换算,冲突则该题作废。

### 2.4 反极性防护（创新①的★）——纯共现的致命陷阱

纯共现的死穴是：`"device B does NOT reach 30% PCE"` 会被误连成"B 的 PCE 是 30%"的正向边。系统在建边前做**句级极性检测**，置信度 `confidence = 共现范围基分 × 极性惩罚`：

- ✅ 正向断言 → `polarity=positive`、高置信 → 可作正向证据；
- ⛔ 否定/对比/假设 → 降权；并且**"规则补全（LLM 没断言、纯靠标签规则补的）的非正向边，直接不建**"。

> 效果：被否定的句子进不了正向证据层，下游 `evidence_chain` 取值时只认 `is_evidence_eligible`（正极性 + 置信达标）的边，从根上避免"把被否定的结论当成事实出题"。

### 2.5 这个基座"合理吗"？——对照 §1 的需求自检

- 要可追溯？✅ 每点带地址，`contains` 链回原文。
- 要可验证？✅ 数值是 `Value`、单位是 `Unit`、条件是 `Condition`，类型显式，验证器能逐项核对。
- 要支撑跨段/跨文献？✅ 横向共现 + `same_as` + `references` + `aligns_with` 提供了跨 chunk / 跨模态 / 跨文献的物理通路。
- 要抗幻觉？✅ 边语义=查表、边存在=物理事实、被否定边降权或不建。

> **衔接**：基座只提供"事实与关系"，但它**不知道一道好题该长什么样**。把事实组织成"一道有逻辑、够难、可回答的题"，是 §3 语义规划层的工作。

## 3. QA 如何建立：先 Q 还是先 A？（创新②，核心）

### 3.1 直接回答："既不是先 Q、也不是先 A，而是先规划、后联合落地"

很多方法要么"先写问句再去找答案"（容易问出文中答不了的题），要么"先抓一段话再编个问题"（容易伪多跳）。我们都不是。我们的顺序是：

```
先定题型逻辑(角色 schema)  →  在 G0 上规划出一个"子图 + 角色分配"(QuestionPlan)
                          →  子图一旦确定，问题的"问法"和答案的"证据"就同时被锁定
                          →  LLM 仅做一次"把这份规划用自然语言说出来"(Q 和 A 一起生成)
```

也就是说：**Q 和 A 在"规划阶段"就被同一份证据共同决定（co-determined），LLM 只是最后把它们一起语言化。**

**为什么这样？** 因为答案的证据（`required_nodes` = G0 里的真实物理节点）在写任何文字之前就被钉死了：

- 答案只能由这些节点的原文 span 支撑 → **可追溯、抗幻觉**；
- 难度由这些节点的结构跨度决定 → **难度真实、防伪多跳**；
- 角色齐全才放行 → **问题逻辑完整**。

下面是规划器 `OverlayPlanner.plan_one` 的 7 步（每步配例子）。

### 3.2 第 1 步：问题逻辑规划

按题型取出角色 schema，确定"这道题需要哪些角色、答案形态是什么"。

> 例（比较题）：目标 = 填满 `object_A/B · shared_attribute · value_A/B · evidence_A/B`，答案形态 = "比较结论（含对象、指标、对齐条件）"。

### 3.3 第 2 步：种子初始化（**不是按标签随机选**）

种子是一道题的"起跑点"——后面所有扩展、角色分配都围着它展开，所以选得好坏直接决定题目质量。

**先回答你的疑问：是不是"拿到种子标签就随机抽一个同标签的点"？不是。** 同标签的点确实很多（比如比较题的种子标签是 `Attribute`，全图可能有上百个 AttributeNode），但系统**不随机**，而是对这一堆同标签候选做"**质量过滤 → 多特征打分 → 排序 → 按序逐个试**"。完整四步如下：

**① 圈定候选池（这一步才用到"标签"）。** 题型决定种子的**候选标签集** `seed_types(qtype)`，把全图中所有该类型的节点收进候选池。

> 例（比较题）：候选标签集 = `{Attribute}` → 候选池 = 全图**所有 AttributeNode**（PCE、Voc、bandgap、stability…，可能上百个）。
> 注意候选标签集本身是**题型专属**的（见下表），所以"用哪类点起步"第一层就由题型逻辑卡死了，不是任意标签。

**② 质量门过滤（先扔掉不配当种子的点）。** 候选池逐个过 `node_ok`（即 `is_valid_point`）：**符号噪声名、停用词、纯通用词、元数据点一律剔除**。这样像表格里 `p / n / b` 这种单字符行标签、`Parameters / No.` 这种通用列头，根本进不了种子竞争。

**③ 多特征打分 SeedScore（核心——这才是"除了标签之外"的选取依据）。** 对每个幸存候选算一个加权分，**同标签的点靠这个分被区分开**：

```
SeedScore(v) = 1.0·领域重要性 + 1.0·证据丰富度 + 0.8·层级权重
             + 0.5·欠覆盖度 + 0.6·专家优先 + 0.5·历史成功率 − 1.0·歧义风险
```

七个特征各自的含义与算法（**这些就是"除标签外的其它选取方法"**）：

| 特征 | 怎么算（实现口径） | 直觉 |
|---|---|---|
| **领域重要性** domain_importance | 带 `domain_schema_tag`（领域 schema 标记）→ 1.0，否则 0.3；给了领域词表则取表值 | 领域核心指标 > 路人词 |
| **证据丰富度** evidence_richness | `min(1, 出入度/8)`——邻居（连着的条件/数值/图表）越多越高 | 周围证据越多越能撑起一道题 |
| **层级权重** node_level_weight | 查表：Claim 1.0 > Method 0.95 > Concept 0.9 > Attribute 0.7 > Value 0.3 > Unit 0.1 | 高层点更适合当复杂题起点 |
| **欠覆盖度** under_coverage | 外部传入（默认 0）：该点/该区域此前被出题覆盖得越少越高 | 鼓励覆盖冷门区域，避免扎堆 |
| **专家优先** expert_priority | 外部传入（默认 0）：可由领域专家/先验注入偏好 | 允许人工把关键点提权 |
| **历史成功率** historical_success | 来自长期记忆 `seed_boost`：该种子过去成功出过好题就加分 | 越用越准（见 §4.3） |
| **歧义风险** ambiguity_risk（**减分**） | 空名/符号名/停用词 → 1.0 重罚；名字 <4 字 → 0.5；否则 0 | 名字模糊的点当种子易产生歧义题 |

> 也就是说，**在所有 AttributeNode 里**，`PCE`（领域核心指标、邻居多、层级 0.7、名字清晰、历史常成功）会被排到很前；而某个只出现一次、名字含糊的属性会沉底。**区分同标签点的，是这 7 个结构/语义/历史特征，不是骰子。**

**④ 排序 + 缓冲队列 + 按序逐个试（不是只取第 1 名）。** 把打完分的候选**降序排列**，取前 `seed_pool`（默认 = `per_type × 4`）个组成**有序种子队列**；然后**从队首开始逐个**喂给 `plan_one` 去规划，**成功一个计一个，直到该题型攒够 `per_type` 道题**为止。

> 为什么要"缓冲 ×4 + 按序试"而不是只用 top-1？因为排第一的种子未必能规划成功（可能扩展后凑不齐最小角色集、或过不了双重多跳）。**排名只决定"尝试顺序"，真正能不能成题还要靠后面 §3.4–§3.8 的硬门槛**。失败的种子被跳过，换下一个，既保证质量又保证产量。

**题型专属种子标签集（①里用的候选标签，决定从哪类点起步）：**

| 题型 | 候选种子类型 | 起步逻辑 |
|---|---|---|
| 比较 comparative | **Attribute** | 先锚定共享指标，再找可比对象 |
| 机制 mechanism | Method / Claim | 先定方法或结论，再扩到中间机制、结果 |
| 跨文献 cross_paper | Concept / Attribute | 先定对齐对象，再找他刊实例 |
| 数值 numerical | Attribute / Value | 先定指标/数值，向上找对象、向下找单位条件 |
| 条件 condition | Claim / Condition / Attribute | 先定结论或条件，再找适用范围与证据 |
| 公式 formula | Equation | 先定公式，再找其推导量与适用条件 |
| 图表 table | Figure / Table / Claim | 先定图表或正文结论，再经图注/引用扩证据 |
| 多跳 multi_hop | Concept / Claim / Method | 从高层节点出发跨 chunk/section 扩多步链 |
| 综述 summary | Section | 以章节为种子，聚合其下 Claim/Concept |

> **走查例（比较题）**：候选池=全部 AttributeNode → 过质量门（扔掉 `No./Parameters` 等）→ SeedScore 打分，`PCE` 因"领域核心+邻居多+名字清晰+历史成功"排进 top-`seed_pool` → 作为队首种子先试。
> 它随即被定为 `shared_attribute` 角色（种子先占其最匹配的主角色），**"先有指标这个锚、比较题才有共享维度"**——正是 §1.1"先定角色"的落地。若 `PCE` 这一支最终凑不齐 `value_B` 而失败，队列里的 `Voc`、`bandgap` 会被依次顶上。

### 3.4 第 3 步：子图扩展（保证"语义完整"与"出题充分"的关键）

围绕种子，在 G0 上**只读地**做三种走法（`expand_candidates`），形成候选节点池：

1. **向下游走**（取完整证据链）：高层 → Concept/Method → Attribute/Condition → Value/Unit → Figure/Table → Evidence。
   > 例：从 `PCE` 向下 → `device A 的 23.5` → `%`，证据链补齐。
2. **向上定位**（确定语义归属，防数值悬空）：Value/Unit → Attribute → Concept → Chunk → Section → Paper。
   > 例：从 `23.5` 向上 → `PCE` → `device A` → `chunk-007`，确认"这个 23.5 是 device A 的、在 results 节"。
3. **同 chunk 邻居 + 二级展开 + 媒体补块**：物理图里跨类型边稀疏（公式节点常孤立、Section 只连 Chunk），所以额外把"到达的 ChunkNode 里的 Claim/Value/Condition"一并拉进来；孤立的 Equation/Figure/Table 则补入**同章节正文 ChunkNode**作其证据邻域。

候选池再按节点类型分桶成 **RolePool(q)**（candidate_objects / candidate_values / candidate_conditions …），后续虚拟边只在池内建立，**缩小搜索空间、提高与题目的相关性**。

> **这一步如何"保证语义完整 + 出题充分"**：
> - 向上定位保证每个值都有归属（语义完整，不会问出"23.5 是什么"这种悬空题）；
> - 向下游走 + 同 chunk 拉取保证证据链和约束（单位/条件）都在场（出题充分，能满足验证器对数值/单位/条件的核对）；
> - 媒体补块保证图表/公式题不会因节点孤立而无证据可引。

#### 3.4.1 校准一下你的理解：候选池怎么选、不满足 schema 会怎样

你的心智模型大体是对的（"同标签池逐一选种子 → 拉邻居进候选池 → 从候选池选需要的点"），但有**三处要校准**，尤其是最后那个关键问题——**不满足基础 schema 不会"直接接收"，而是先补、补不上就整题丢弃**。把完整链路摆清楚：

**校准点 1：候选池不是"种子的直接邻居"，而是"种子的多跳证据邻域"。**
拉进候选池的不是一圈直接相连的点，而是 §3.4 三种走法（向下游走多跳、向上定位到 Paper、同 chunk + 二级展开 + 媒体补块）得到的**一整片证据邻域**（实现上限约 80 个节点）。这样才可能一次性把"对象、指标、值、单位、条件、证据 span"都囊括进来。

**校准点 2：从候选池"选需要的点"不是裸取，要过三道"质量/语义关"，不是见到合适类型就接收。**

1. **类型约束**：每个角色只接受特定 NodeType（`value_A` 只能是 ValueNode、`shared_attribute` 只能是 AttributeNode…），类型不符根本不进槽。
2. **质量门**：给高层角色（Concept/Attribute/Method/Claim）填点时再过一次 `node_ok`，符号噪声/通用词点被挡掉。
3. **语义绑定**（§3.5）：光"类型对"还不够——比较题会沿物理边把 `value_A` 绑回 `object_A`（表格路径或 `has_attribute→has_value` 链），**确保这个值真的属于这个对象**，而不是同类型里随便抓一个。

**校准点 3（你最关心的）：基础 schema（最小角色集）若没填满，绝不"直接接收"，而是分三段处理——补 → 接地 → 丢弃。**

```
角色齐了吗？
   ├─ 缺角色 → ① 虚拟边补全(§3.6)：针对缺失角色造候选虚拟边，
   │             但每条必须能在 G0 找到物理回落路径 + Score_q>0 才被接受，否则不算数
   │          ② 证据角色接地(§3.7)：仍空缺的 evidence_* 角色回落到锚点所在 ChunkNode
   ├─ 补完仍缺最小角色集 → ★ 整道题作废(plan_one 返回 None)，并写入“失败记忆”
   └─ 齐了 → 进入子图定形 + 硬门槛验收
```

也就是说，**"未满足 schema 就直接接收"在本系统里不存在**。最小角色集是一道 `role_completeness ≥ 0.99` 的**硬门槛**（比较题必须凑齐 `object_A/B · shared_attribute · value_A/B · evidence_A/B`，缺一个都不行），凑不齐的种子直接被放弃，换队列里下一个种子重来。

**校准点 4：候选池里确实"还有一轮打分排序"，而且是和种子那轮不同的第二轮打分。** 系统里有**两轮独立打分**，别混淆：

| 打分轮次 | 打什么分 | 选什么 | 在哪一步 |
|---|---|---|---|
| **第一轮 · SeedScore** | 给同标签候选打分 | 选**起跑种子**（唯一） | §3.3 |
| **第二轮 · Accept(v) 综合评分** | 给候选池里的点打"纳入子图能不能让整体 Utility 提升"的边际增益 | 选**进入最终子图的点** | §3.7 |

第二轮的规则是 **Accept(v)**：以"已分配角色的节点 + 种子"为基底，候选池里的点**逐个试纳入**，只有当它能让综合评分 `Utility` 的增益 `> τ` 时才真正保留，否则丢掉——**所以候选池绝不是照单全收，而是"边算边收，只收让题更好的点"**。综合评分公式见 §3.7。

> 一句话总结这条链：**同标签池 →(SeedScore 打分排序)→ 唯一种子 →(多跳邻域)→ 候选池 →(类型+质量+语义绑定)→ 填角色 →(缺角色就虚拟补全/接地，补不上就整题丢弃)→(Accept(v) 第二轮打分)→ 定形子图 →(硬门槛验收)→ 成题或作废。** 每一环都有门槛，没有"凑合接收"。

#### 3.4.2 从单个种子到"第一张骨架子图"：它是**物理的**，不是虚拟的

这是最容易误解的一点，先把结论摆正：**最开始的骨架子图不是用虚拟边搭的，而是完全用 G0 的真实物理边长出来的。** 顺序是"**先用真边搭骨架 → 缺角色才补虚拟边**"，不是反过来。`Gq = G0[Vq] ⊕ Ωq` 里，主体是 `G0[Vq]`（真实诱导子图），`⊕ Ωq`（虚拟边）只是后补的补丁层。

**单个种子凭什么能长出骨架？** 因为种子在 G0 里**本来就带着真实邻接**——§2 建图时已经把 `has_attribute / has_value / has_unit / under_condition / contains` 这些**带语义类型的真实边**连好了。从一个种子 `PCE` 出发，沿这些真边走（§3.4 的向下/向上/同 chunk），就能摸到一圈彼此语义相关的真实点：

```
顺真实边游走（全部是 G0 里已存在的物理边）：
   PCE ──has_value──▶ 23.5 ──has_unit──▶ %          （device A 行 ∩ PCE 列）
   PCE ──has_value──▶ 21.x                            （device B 行 ∩ PCE 列）
   PCE ◀─has_attribute── device A / device B          （向上定位到对象）
   23.5 ── under_condition ──▶ 85°C aging             （拉到约束）
   device A/B ── 所在 ChunkNode/Table                  （拉到证据 span）
```

把这些真实点按角色摆进槽位，就得到**第一张骨架**。它的"脊梁"`base` = 种子 + 所有已分配角色的节点（即 §3.7 里 `_select_subgraph` 的 `base`）。**这一步里一条虚拟边都还没有。**

**怎么保证这张骨架"有基本语义、能直接被子图打分"？** 不靠事后补救，而靠三件在搭骨架时就已成立的事：

1. **角色 schema 本身就是语义骨架**：题型规定了哪些槽必须有（object/attribute/value/evidence…），打分项 `role_completeness` 直接数"填了几个槽 / 最小角色集"——骨架天生对着题型语义，分自然打得出。
2. **拉点走的是"带语义的真边"**：扩展只沿 `has_value/has_attribute/under_condition` 这类**已带语义类型**的物理边游走，所以进骨架的点彼此语义相关，不是散点堆。
3. **结构连通性可计算**：打分项 `structural_coherence` 用**这些节点之间的真实 G0 边**判断它们是否连成一张通图——骨架的边都是真的，连通度算得出，`utility_score` 从第一张骨架就能跑。

> 也就是说：**子图语义打分评的全是"真节点 + 真边 + 填了几个角色"这些已经物理成立的量**，所以哪怕只有一个种子起步，第一张物理骨架也立刻可打分、可验收。虚拟边只有在"物理骨架缺某个必需角色"时才被请出来补缺（§3.6）。

### 3.5 第 4 步：角色分配 + 语义绑定（让数值"真正属于"对象）

先**物理优先**地把候选节点按类型填进角色槽（种子先占它最匹配的主角色）。但"类型匹配"会犯一个错：把任意一个 Value 填进 `value_A`，导致**数值不属于被比较对象**。于是加一层**题型专属语义绑定**：

- **比较题**：沿物理边把值绑回对象——
  - 表格路径（本语料主导）：对象=表行，指标=两行**共享的列**，值=该行该列**单元格**；
  - 通用链路：`object_A —has_attribute→ 共享指标 —has_value→ value_A`。
  - 单位绑定为该数值真正的 `has_unit`，或表列头的 `has_unit`（杜绝游离噪声单位如 `unit='a'`）。
- **跨文献题**：用 `same_as / aligns_with` 对齐对锚定 A/B 实例，且强制 `paper_B_instance` 与 `paper_A_instance` 来自**不同 paper_id**（否则整题作废）。**并过条件兼容门**：两实例的 `under_condition` 同维度须同值或可换算,条件冲突（如 A 在 85°C、B 在常温）则**整题作废**——杜绝把"不同条件下的两个值"包装成可比题。

> 例（比较题）：绑定后确认 `value_A=23.5(device A 行 ∩ PCE 列)`、`value_B=device B 行 ∩ PCE 列`，
> 单位统一取 PCE 列头的 `%`。**两个值真正可比，且都挂在 PCE 上**——这是"比较"成立的物理保证。

### 3.6 第 5 步：核心推断边 + 虚拟边补全 Ωq（虚拟边 = 被推断出的关系）

**先重立"虚拟边到底是什么"——它是一条"被推断出来、但任何单一原文 span 都没有直接陈述"的关系。** 实体点早在 §2 就抽好了,虚拟边**不创造任何新点**;它也**不是某条物理边的换皮**。它表达的是题目要求答题者**做出的那一步推断**——例如"device A 在 PCE 上 > device B"这条比较关系,**不存在于任何一句话里**,必须被 posit(假设)出来。这正是难题的难度来源。

一条虚拟边由六个要素界定（五元组 + 推断算子）：

```
虚拟边 e = ⟨ source, target, virtual_type, question_role, inference_op, backing_path ⟩
```

- **source（锚点）**：一个**已分配好的角色节点**（如比较题里已定的 `object_A`）。
- **target（候选）**：能填上缺失角色 / 充当推断另一端的真实候选点（如 `object_B`）。
- **virtual_type（规划意图）**：规划期"怎么搜"的意图,取自固定枚举（`SEARCH_COMPARABLE / LIMITED_BY / SEEK_MECHANISM / CROSS_PAPER_ALIGN / NEED_VISUAL_EVIDENCE` …）。
- **question_role（搭载角色）**：这条边为填补哪个角色而生。
- **inference_op（推断算子，★新增）**：答题者要做的**推理操作**,取自 `InferenceOp` 枚举（`COMPARISON / CONDITION_SHIFT / CAUSAL_CHAIN / CROSS_PAPER_ALIGN / AGGREGATION / VISUAL_GROUNDING / ALIAS_BRIDGE / ATTRIBUTE_LOOKUP / CONDITION_BIND / CONTRAST`）。**它决定题目的推理难度下限（§1.2）**,也决定 backing 的路由方式。
- **backing_path（凭据,★重定义）**：**不是这条关系本身,而是它的"可证伪性凭据"**——在 G0 上为每个操作数找到的真实物理路径。如比较关系"A>B on PCE"的凭据 = 两条子路径 `A→PCE→23.5` 与 `B→PCE→21.8`,**而非一条 A→B 的捷径边**。

> 所以"虚拟边如何界定"的新答案是：**它 = 一条被推断出来、单 span 不存在的关系,携带一个推断算子,其每个操作数都必须能在 G0 上找到物理凭据路径。** 三个"必须"——必须是推断(单 span 不存在)、必须对应一个推断算子、必须每个操作数都能物理回落——把它和"换皮的物理边"以及"随便加一条边"彻底区分开。

#### 3.6.0 核心推断边 always 铺设(把虚拟边提升为主干)

每个题型声明一个**核心推断算子**与对应的**核心虚拟边模板**(`core_inference(qtype)`)。规划时,**只要角色绑定完成,核心推断边就被铺设——即使最小角色集已被物理扩展填满也照建**(标 `is_core=true`)。这把虚拟边从"事后补缺的补丁"变成"每道题的推理主干"。

| 题型 | 核心 InferenceOp | 核心虚拟边 |
|---|---|---|
| comparative | COMPARISON | `object_A —COMPARABLE→ object_B`（锚在 shared_attribute） |
| condition | CONDITION_BIND | `claim —LIMITED_BY→ condition_boundary` |
| mechanism | CAUSAL_CHAIN | `method —SEEK_MECHANISM→ mechanism —METHOD_EFFECT→ result` |
| cross_paper | CROSS_PAPER_ALIGN | `paper_A_instance —CROSS_PAPER_ALIGN→ paper_B_instance` |
| table | VISUAL_GROUNDING | `text_claim —NEED_VISUAL_EVIDENCE→ figure/table` |
| numerical/formula | ATTRIBUTE_LOOKUP | 单跳事实,不强制核心边,仅缺角色时补全 |

> **关键:表格物理路径不再"绕过"虚拟边,而是成为核心边的凭据。** 表格比较题里,§3.5 已定位的两条单元格物理路径,正好被登记为核心 COMPARISON 边的 backing_path——于是核心虚拟边**不是空壳**,而是"以真实物理证据为凭据的被推断比较关系"。

#### 3.6.1 缺失角色的补全边(原虚拟边逻辑,退居其后)

核心边之外,若某个必需角色在 G0 里物理上仍够不到(如比较题只找到 device A、缺 `object_B`),再**针对缺失角色**提补全候选虚拟边——每条仍搭载 `question_role`,按题型意图生成：

- 比较缺对象 → `PCE —SEARCH_COMPARABLE→ device_B`（搭载 `object_B`）；
- 条件题缺边界 → `claim —LIMITED_BY→ 某 Condition`（搭载 `condition_boundary`）；
- 机制题缺中间机制 → `method —SEEK_MECHANISM→ 某 Claim`；
- 跨文献缺他刊实例 → `canonical —CROSS_PAPER_ALIGN→ 他刊节点`；
- 任意题缺证据 → `anchor —NEED_VISUAL_EVIDENCE→ Figure/Table`。

每条候选边(核心边与补全边同)要过**两关 + 一门**：

1. **证据可回落（硬条件）**：`find_backing_path` 为每个操作数找物理凭据路径。**凭据路径只能走可作证据的正向边**（极性为正、置信达标——被否定的共现边不得作凭据）；找不到 → `rejected`。
2. **条件兼容门（硬门槛,★新增；仅 COMPARISON / CROSS_PAPER_ALIGN）**：两操作数的 `under_condition` 同维度须同值或可换算,**冲突即整边作废**（杜绝伪可比题）；一端有条件一端无 → 标 `weak` 降权。
3. **问题级打分** `Score_q(e)`：

```
Score_q = 1.0·目标相关 + 0.8·语义相似 + 1.0·schema兼容 + 1.2·物理可回落(最高权重)
        + 0.8·证据多样性(跨chunk/跨paper) + 0.6·难度增益 + 0.7·条件兼容(由条件兼容门给分)
        − 1.0·捷径风险(同chunk重罚) − 1.0·歧义风险
```

> **反捷径的硬化**：对**核心边**且目标难度为 L3/L4 时,凭据路径若触及的 chunk 少于 2 个 → **直接硬拒**（不只是 Score_q 扣分）。这把"双重多跳"从软惩罚升级为硬约束。
>
> **与基座的联动**：虚拟边连接的两端**仍是 G0 里的真实节点**；它必须能为每个操作数找到物理凭据路径才生效；它只写进 `semantic_overlay_edges`（规划用），而 `required_nodes / required_evidence_paths` **永远是 G0 物理节点与物理路径**。脚手架（Ωq）决定怎么盖这道题、要求做哪一步推断,但每根都钉在地基（G0）上,盖完即拆,地基不动。

#### 3.6.2 实现现状（诚实说明）

✅ **本节描述的是改造后的目标实现：核心推断边 always 铺设、虚拟边主导规划意图、物理图退为"找凭据落地"的底座。** 消融开关 `use_core_edges` / `use_overlay` / `use_compat_gate` 可逐个关闭,用于度量"虚拟边主干化""核心推断边""条件兼容门"各自的边际贡献。关闭 `use_core_edges` 时即退化为早先的"物理优先 + 仅缺角色才补虚拟边"的保守路线,作为对照基线。

### 3.7 第 6 步：综合评分 + 难度封顶 + 子图定形

用 **Accept(v)** 增量地选定最终子图：从"已分配角色的节点 + 种子"为基底，逐个尝试纳入候选节点，**只有能让综合评分提升（增益 > τ）的节点才保留**——避免把无关噪声塞进子图。综合评分：

```
Utility = 0.8·语义 + 1.5·角色完整 + 1.3·证据充分 + 1.0·约束覆盖
        + 0.7·结构连通 + 0.8·难度匹配 − 1.2·捷径惩罚 − 0.8·噪声惩罚
```

#### 3.7.1 子图本身是怎么打分的（八个分项怎么构成）

`utility_score` 对**整张当前子图**算一个分，八个分项全是**确定性、纯 Python**的量（无 LLM、无向量模型）：

| 分项 | 怎么算（实现口径） | 评什么 |
| --- | --- | --- |
| **语义** semantic | 见下方 §3.7.2，theme-token 重叠率 | 子图整体是否扣住种子主题 |
| **角色完整** role | 填满的最小角色数 / 最小角色总数 | 题型逻辑是否成立（权重最高 1.5） |
| **证据充分** evidence | 关键角色中"能回到带地址物理节点"的占比 | 答案是否可追溯 |
| **约束覆盖** constraint | `{数值, 单位, 条件}` 三类命中几类 / 3 | 数值/条件题是否够料 |
| **结构连通** structure | 子图节点间**真实 G0 边**的最大连通分量 / 节点数 | 是逻辑链还是散点 |
| **难度匹配** difficulty | (跨 chunk 数, 证据 span 数, 路径长) 对目标难度要求的达成度均值 | 难度是否名副其实 |
| **捷径惩罚** shortcut（减） | L3/L4 却只占 ≤1 个 chunk → 重罚 1.0 | 反伪多跳 |
| **噪声惩罚** noise（减） | 子图里符号/低质量节点占比 | 反垃圾点 |

#### 3.7.2 "图的语义分"具体怎么构建（你问的重点）

语义分**刻意做得很轻**——不引入词向量/嵌入依赖，而是用 **canonical_id 命中 + theme-token 重叠**近似：

1. 取**主题 theme** = 种子节点的规范名（如 `PCE / power conversion efficiency`），并取其 `canonical_id`；
2. 对子图里每个有内容的节点：**先比 `canonical_id`**——与主题同 `canonical_id` 即命中（这一步免费解决同义异形,`PCE` 与 `efficiency` 若共享 canonical_id 直接算命中）；**否则回退**到名字/内容 token 与 theme-token 是否至少共享一个；
3. 语义分 = 命中节点数 / 有内容节点数；
4. 主题为空时回退到中性分 0.5。

> 例：theme=`PCE`(canonical_id=`pce`)，子图里 `efficiency under aging`(同 canonical_id `pce`)即便表面 token 不含 "pce" 也**经 canonical_id 命中**；某个无关的 `crystal size` 既不同 canonical_id、token 也不重叠 → 不命中。
>
> 为什么先用 canonical_id 再退 token 重叠而不用嵌入？**为了零依赖、可复现、够快**——子图打分要在贪心里被反复调用，必须便宜（见 §3.7.4 性能）；而 canonical_id 是 §2 建图时已经算好的,复用它即可解决同义异形,无需新依赖。语义只占 0.8 权重、且是软指标。
>
> ⚠️ **诚实说明（别被名字误导）**：即便加了 canonical_id,这个"语义分"本质仍是**对齐/标签级**的命中率,**它并不承载"整道题在问什么"的语义**。它的真实作用只有一个：**当软性正则项，把明显跑题的点压低分**。
>
> **那"整道题的语义"到底落在哪？落在三处,都不是这一项**：① **关系结构语义**——题意是角色 schema 关系模板（题意是*关系*，不是*词*）；② **推断算子语义**——核心虚拟边携带的 `inference_op`(COMPARISON / CAUSAL_CHAIN …)明确标出"这道题要做哪一步推理"（§3.6）；③ **物理证据语义**——每个角色/操作数回落到带地址的 chunk/value/condition。所以单看 `semantic_score` 作用有限,真正的语义承载在角色关系 + 推断算子 + 物理证据上（详见 §3.7.5）。

#### 3.7.3 候选池有预排序吗？是动态打分还是只打一次？（直接回答）

**两件事要分清——一个静态、一个动态：**

- **候选池预排序：静态，只排一次。** 进入贪心前，候选池按 `_cand_priority`（**按节点类型的固定优先级**：Condition 0.95 > Value/Evidence 0.9 > Unit 0.85 > Figure/Table 0.7 > Claim/Attribute 0.6 > 其它 0.4，带 chunk_id 再 +0.05）**降序排一次**，得到尝试顺序。这一步**不随子图变化**，算一遍就固定。
- **子图综合评分：动态，每试纳入一个候选就重算一次。** 贪心按预排序顺序逐个取候选 `c`：构造 `trial = 当前子图 + c` → **对整张 trial 重新算一次 `utility_score`** → 若边际增益 `score(trial) − score(当前) > τ` 就保留 `c`、更新当前分，否则丢弃 `c` 继续下一个。

所以答案是：**不是"对候选集整体只打一次分然后取 top-k"，而是"候选先按静态优先级排好序，再沿这个序做贪心前向选择——每纳入一次就对整张子图动态重算一次综合分，只收能让整体变好的点"。** 静态预排序决定**试的顺序**，动态增量打分决定**收不收**。

> 直觉：静态优先级先把"高价值类型"（条件/数值/证据）排到前面优先尝试；动态增量分再保证"即使类型高价值，若它和当前子图凑不成更好的逻辑结构，也不收"。两者配合，既快又不贪坏。

#### 3.7.4 "逐一尝试"会不会慢？——不会，因为评分极廉价且有三重封顶

会慢的直觉来自"每个候选都要重算一次子图分"，但实际开销可忽略，原因有三：

1. **每次打分是纯 Python 微秒级**：八个分项都是小集合上的 token 重叠 / 计数 / 子图内 BFS 连通，**没有任何 LLM 调用、没有嵌入、没有磁盘**。被反复调用的正是这种廉价函数。
2. **规模三重封顶**：候选池本身上限约 80 个点（`expand_candidates` 的 `max_total`）；最多只接受 `max_expand=12` 个额外点（到顶即停止尝试）；子图节点数始终是几十量级。所以一道题的贪心最多约 80 次"小子图重算"，全是 CPU。
3. **真正的成本在别处**：整条流水线唯一的重负载是 **LLM 调用**，而 LLM **每道成题只在 generate 阶段调用一次**（规划阶段 0 次 LLM）。子图贪心选择相对 LLM 几乎不计时间。

> 量级对照：每题贪心 ≈ 数十次纯内存评分（亚毫秒级累计）；而每题 1 次 LLM 生成 ≈ 数秒。**贪心选择不是瓶颈，省掉它也省不下可感知的时间，却会让子图质量下降**——所以这是一笔很划算的"用廉价 CPU 评分换题目质量"的交易。
>
> 另：`use_score_guided=false` 是消融开关——关掉贪心就退化为"按静态优先级直接取前 `max_expand` 个"，用于度量 Accept(v) 的边际贡献，正式产线默认开启。

子图定形后，难度先由结构变量算出，再**按真实跨 chunk 数封顶**：单 chunk 子图最高只能是 L2，2 个 chunk 最高 L3，≥3 个才 L4。

> 例：若 device A、device B 的证据落在同一 chunk，哪怕节点很多，也会被从 L4 封到 L2——**杜绝"富节点单段"被误判为高难度**。

#### 3.7.5 澄清：一道题只用一个种子，但"语义中心"不等于那一个种子

你的担忧很合理——如果整张子图的语义都只系在一个种子上，那中心会很脆弱。但实际不是这样，要把三件事分开看：

**① "一个种子"是"每道题一个"，不是"整个题库一个"。** 一道 QuestionPlan 确实只围绕**一个**种子展开（`plan_one(qtype, seed_node)`）；但每个题型会从排好序的种子队列里扫 `seed_pool ≈ per_type×4`（默认约 48）个种子，**每个种子各生成一道题**。所以一个题型下有几十个不同的语义中心，整个 benchmark 建立在**大量种子**上，绝不是"一个种子撑全场"。

**② 单道题的语义中心是"角色星座"，种子只是其中的锚桩。** 种子只占住该题型**天然的枢纽角色**（比较题=`shared_attribute`、机制题=`method`、跨文献=`canonical_concept`），其余角色（`object_A/B`、`value_A/B`、`evidence_A/B`…）是**靠物理扩展找来的别的真实节点**填的。真正定义"这道题在问什么"的，是**整组被填满的角色 + 它们之间的真实 G0 边**，而不是孤立的种子。

> 例（比较题）：种子 `PCE` 只是枢纽指标，device A、device B、23.5、21.x、各自 chunk 都是别的点。把它们用 `has_value/has_attribute` 真边绑成一张星形结构——**这张星座才是语义中心，PCE 只是星座的中心锚**。而且这种"以共享指标为枢纽"正是比较题该有的样子（§3.5 的语义绑定就是在强化这个枢纽），所以**单锚点不是缺陷，而是贴合题型逻辑的设计**：每个题型恰好有一个天然枢纽，硬塞多锚点反而会模糊问题焦点。

**③ 子图分的"迭代更新"不是只靠种子——8 个分项里只有 1 个依赖种子文本。** 你看到的"语义相似度只依托种子主题"只说中了 `utility_score` 的**一项**：

- **唯一与种子文本挂钩的是"语义分"**（theme-token 重叠，权重 0.8，软指标）——它衡量"子图是否扣住种子主题"，确实以种子规范名为 theme。
- **其余 7 项与种子文本无关，评的是整张子图自身**：角色完整（1.5，数填了几个角色）、证据充分（1.3，看能否回落物理地址）、约束覆盖（1.0）、结构连通（0.7，用整图真实边算连通分量）、难度匹配（0.8）、捷径惩罚（−1.2）、噪声惩罚（−0.8）。

> 所以贪心里每纳入一个点触发的整图重算，**主要由角色/证据/结构这些"整图属性"驱动，种子主题只贡献一个软信号**。即便语义分这一项的 token 近似漏掉了同义词（如 `PCE` vs `efficiency` 未共享 token），role/evidence/structure 仍会把真正成链的点留下、把散点挡掉——硬保证不落在种子身上。

**一句话**：种子是"起跑锚"和"主题软提示"，**不是整张子图语义的唯一来源**；语义中心是被角色 schema 组织、由真实物理边连成的整组节点，benchmark 则由跨题型、跨大量种子的扫描共同覆盖。

### 3.8 第 7 步：逻辑充分性 + 双重多跳验证 → 输出 QuestionPlan

子图必须同时过两道**硬门槛**才放行：

- **逻辑充分性**：角色完整度 ≥ 0.99（最小角色集全满）、证据可回落 ≥ 0.6、结构连通 ≥ 0.5、难度匹配 ≥ 0.5、捷径惩罚 ≤ 0.5；**约束覆盖 ≥ 0.5 仅对数值/公式/比较/图表题强制**（避免一刀切卡掉本不含数值的合法题）。
- **双重多跳**（防伪多跳）：L3 需 ≥2 个 chunk 且语义路径 ≥2；L4 需 ≥3。隐含**证据遮蔽测试**思想——去掉关键证据就答不出、单 chunk 答不了 L3/L4，否则降级。

通过 → 沉淀正向记忆并产出 `QuestionPlan`，其中包含：`required_nodes`（答案证据，全是 G0 物理节点）、`role_assignment`、`required_evidence_paths`、`constraints`（数值/单位/条件，单位与各自数值精确绑定）、`forbidden_shortcuts`（L3/L4 禁止单片段作答）、`forbidden_generalization`（有条件就禁泛化、有单位就禁改单位）、`inference_ops`（本题核心推断算子）、以及 `masking_spec`（★遮蔽测试规格）。失败 → 记入失败记忆并丢弃。

> **遮蔽规格 `masking_spec`（把双重多跳从"结构代理"升级为"行为契约"）**：对每条核心推断边记录"删掉哪个操作数的凭据子路径应导致不可答"——`drop_operand`=[操作数 A 的凭据节点]（删 A 应使题不可答）、`drop_cross_chunk`=[跨 chunk 的桥接节点/边]（限单 chunk 应使 L3/L4 不可答）。这份规格被 §4.1 验证阶段的**遮蔽检查器**消费：用强模型在删除指定证据后重答,若仍答得出 → 说明该题是伪多跳 / 可走捷径,判违规并降级。于是"非伪多跳"不再只是结构条件,而是**可执行的行为保证**。

> 这份 QuestionPlan 就是"问题与答案的联合规格说明"。**问与答此刻已被同一组 G0 证据共同锁定。**

### 3.9 第 8 步：LLM 落地 QA（generate）+ 后过滤

把 plan 的证据 span 打包给 LLM，让它**按约束**一次生成 `{question, answer, evidence_path, constraints}`。LLM 不决定"用哪些证据"（那是规划定的），只负责"把这份规划说成通顺的一问一答"。生成后做**后过滤**：泄漏 node_id、答非数值题却无数字、与原文无 token 重叠、同题型同答案重复——一律丢弃。

同时从 QA + 图谱**直接派生**多类训练语料：evidence-grounded instruction、graph reasoning trace（`节点→节点` 推理链）、RAG grounding、refusal（证据不足时）。

> 例（最终成题）：
> Q：「在 85°C 老化条件下，device A 与 device B 的 PCE 哪个更高，分别是多少？」
> A：「device A 为 23.5%，高于 device B 的 …；该结论限定于 85°C 老化条件。」
> 证据路径：`device A → PCE → 23.5 → %` 与 `device B → PCE → … `，证据 span 指回 chunk-007 / Table 1。

## 4. 自更新与迭代（创新③）

QA 生成后进入"腐蚀 → 验证 → 修复"闭环（对应 corrupt → verify → repair）。

### 4.1 四层一致性验证（verify）

对每条 QA 跑可组合的检查器，不通过就产出**结构化 violation**（带 `field/expected/actual/severity/repair_hint`）：

- **数值层**：答案里的数字必须在证据中出现（允许 2% 容差）。例：答案写 24.0% 但证据是 23.5% → `critical`。
- **单位层**：答案单位与证据单位**同量纲可换算**，否则违规。
- **条件层**：条件型答案必须**显式保留限定条件**，出现 "in all cases / 所有情况" 等泛化措辞且原文有条件 → 违规。例：把"85°C 下 23.5%"答成"任何条件下都 23.5%"。
- **证据层**：答案要有引用且与证据 span 有 token 重叠，否则视为无依据。
- **遮蔽层（★新增，行为级反伪多跳）**：消费 plan 的 `masking_spec`——按规格删除某个操作数的凭据子路径 / 限制为单 chunk 后,把"被遮蔽的证据"喂给强模型重答；若**仍能答对**,说明该题不真正依赖被删证据（伪多跳 / 可走捷径）→ 判违规并降级难度。这把 §3.8 的结构性双重多跳升级为**可执行的行为验证**。
- （可选）**LLM 语义层**：判答案是否忠实于证据、有无外部幻觉。

合法拒答（"文中无法确定"）直接判通过，不误入人审。

### 4.2 违规驱动的自修复（repair）——和"自然语言 self-refine"的本质区别

修复时喂给 LLM 的**不是"你再想想"**，而是带 `field/expected/actual/source_address` 的**结构化违规报告**——提供明确的外部锚点，避免模型自洽幻觉。流程：

```
对未通过的 QA：用 violation 报告局部改写答案 → 重新跑约束层 → 仍不过则下一轮，最多 K=3 轮
```

- 改对了 → 进 `final` 终池；
- 改成合法拒答 → 也算通过（证据确实不足）；
- K 轮仍不过 → 进 `human_review` 池。

全程记录 `wrong → violation → repaired` 修复轨迹，本身就是一类高价值训练数据（corpus/repair.jsonl）。

> **重要边界**：在 QA 层，违规改的是**答案**；在规划层（语义规划方法论），违规报告反向改写的是 **Ωq（虚拟覆盖层），绝不改 G0**。地基永远不动，这样**同一张 G0 可被无数问题安全复用**。

### 4.3 长期记忆（跨运行沉淀）

规划过程把结果沉淀为三类记忆，用于初始化下一轮、让系统越用越准：

- **Overlay Pattern Memory**：成功的"题型 + 角色节点类型链"签名 → 下次同模式优先；
- **Failure Memory**：失败模式 → 规避；
- **Seed / Domain Preference Memory**：成功种子的 `historical_success` 回流进 SeedScore，好起点被持续加权。

## 6. 合理性讨论与边界

- **基座合理性**：G0 只承载"物理事实 + 结构 + 可追溯地址"，边语义靠查表、边存在靠共现、被否定边降权——保证它是一个**可信、可复用、抗幻觉**的固定底座。不合理之处主要来自上游解析噪声（表格 HTML、符号名），已用质量门（`is_valid_point / is_meaningful_attribute`）拦截。
- **出题合理性**：难度锚定结构、角色硬门槛、双重多跳 + 难度封顶——共同保证"难是真难、问有逻辑、答有证据"。
- **已知边界**：物理图跨类型边稀疏（公式/图常孤立），靠 `chunk_id` 邻域和同章节补块兜底；`aligned_condition` 不再仅作软约束——在**比较 / 跨文献题**里它升级为**条件兼容门**（§3.5/§3.6），两操作数条件冲突即整题作废；其余题型仍作为规划约束（写进 `forbidden_generalization`）。改造后虚拟边以**核心推断算子**为主干、物理证据为凭据,难度由**算子下限 ⊕ 结构封顶**双轴决定,非伪多跳由**结构双重多跳 + 行为遮蔽测试**双重保证。

### 一句话总结

> **§1 定义"什么是好题（角色 + 推断算子 + 结构化难度）" → §2 建一个可追溯、分层、抗幻觉的固定基座 G0 来提供事实 →
> §3 用"先角色、再种子、再扩展、再绑定、再铺核心推断边、再条件兼容、再双重验证"把事实规划成一份 Q/A 联合锁定的 QuestionPlan，LLM 仅负责语言化 →
> §4 用结构化违规报告 + 行为遮蔽测试驱动自修复，并把成败沉淀为长期记忆。** 每一步的产出都是下一步的输入，环环相扣，G0 永不被污染。
